# Модель прецедентів

У цьому файлі необхідно перелічити всі документи, розроблені в проекті та дати посилання на них.

Модель прецедентів повинна містити загальні оглядові діаграми та специфікації прецедентів.

# Відношення узагальнення

"Користувач" = User and "Технічна підтримка" = admin.

# Структурування моделі варіантів використання для адміністратора

# Діяльність користувача

# Реєстрація користувача

ID: USR.REG
Учасники: Користувач, Система
Передумови: Обліковий запис користувача в системі відсутній
Результат: Новий обліковий запис

# Авторизація користувача

ID: USER.LOG_IN
Учасники: Користувач, Система
Передумови: Зареєстрований користувач
Результат: Авторизований користувач

# Запит на пошук та аналіз даних

ID: QUERY.SEARCH
Учасники: Користувач, Система
Передумови: Авторизований користувач
Результат: Відповідь на запит на екрані користувача

# Експорт результатів

ID: QUERY.EXPORT
Учасники: Користувач, Система
Передумови: Користувач відправив запит на пошук та аналіз даних
Результат: Файл в обраному форматі з результатами аналізу

# Редагування запиту

ID: QUERY.EDIT
Учасники: Користувач, Система
Передумови: Користувач відправив запит на пошук та аналіз даних
Результат: Вивід нового результату на екран користувача

# Технічна підтримка

ID: SUP.HELP
Учасники: Користувач, Оператор технічної підтримки
Передумови: Користувач має певні проблеми при роботі з системою
Результат: Виконання запиту з допомогою адміністратора

# Виключні сценарії

Виключні сценарії - це ситуації, що суперечать виконанню передумов.

# 1. Відхилення запиту на реєстрацію

ID: USER.REG_EXС

# 2. Відхилення запиту на авторизацію

ID: USER.LOG_IN_EXC

# 3. Відхилення запиту користувача на обрані маніпуляції над обраним запитом

ID: QUERY.EXC
